# 框架：Switch Image Crawler

**功能 ID**：003
**创建日期**：2026-04-18
**状态**：框架通过 — 于 2026-04-18 批准

---

## 问题

SwitchAtlas 的价值主张是"可视化的交换机数据库"，但今天绝大多数开关详情页是**没有图片的**。

具体基线（仓库当前状态）：
- `data/vendors/` 下约 **512 个开关文件夹**（跨 Akko、Cherry、Durock、Gateron、JWK、Kailh、NovelKeys、Outemu、TTC、Zeal）。
- `public/images/vendors/` 下只有 **4 个厂商目录**，且实际只有 **Cherry MX Red 一个开关** 有填充的图片资产。
- 结果：访客打开一个开关详情页（例如 Akko Piano V3 Pro、Gateron Ink Black V2），看到的是规格 + 力曲线，但**缺失产品照**——页面看起来像是 404 或未完成的草稿。

用户今天的变通方法：离开 SwitchAtlas，去 switchesdb.com、lumekeebs.com、milktooth.com、或谷歌图片搜索，目视确认这个开关长什么样，然后才能判断"就是这个"。这直接把信任和流量让给了竞品。

## 受影响的用户群

- **所有详情页访客**——这是最高流量的页面类型（厂商 → 开关）。
- **键盘社区的新手**：他们不知道"Akko Cream Yellow"长什么样，没图无法选购 → 直接流失。
- **有经验的买家对比购买时**：需要并排的视觉确认来区分 V1/V2/V3 Pro、POM vs. Crystal 系列等细微差异。
- **SEO / 站外发现**：无图页面在 Pinterest、Reddit 嵌入、谷歌图片结果中不可见。

战略价值：高。开关详情页是产品的核心单元；没有图就没有"图谱"。

## 商业价值

解决此问题会改变：

- **可信度**：页面看起来完整，不像半成品。与 ThereminGoat 数据 + 力曲线结合后，SwitchAtlas 具备成为"规范来源"的外观基础。
- **竞品对位**：缩小与 switchesdb.com 的功能差距——用户不再需要离站做视觉确认。
- **留存 / 会话深度**：带图的页面更容易让用户从一个开关浏览到下一个（"这个看起来像 X，我也看看 X"）。
- **SEO 长尾**：每个开关页获得可索引的图片资产，扩大搜索可见性。

## 证据

- **定量**：512 个开关文件夹 vs. ~1 个有图片的开关 ≈ **<1% 视觉覆盖率**。这是最显眼的数据缺口。
- **定性**：README 本身已经列出 lumekeebs.com、switchesdb.com、milktooth.com 作为应该整合的来源——说明问题已被意识到但未执行。
- **对比**：switchesdb.com 每个开关都有至少一张产品照，这是该类站点的基础预期。
- **假设**：我们假设这三个站点的图片在归因 + 自托管 + 遵守 robots.txt / 限速 的前提下可用。需要在塑形阶段验证每个站点的 ToS。

## 时间预算

**Medium Batch — 2-3 个会话**

粗略的问题分解（仅为理解范围，非解决方案设计）：
- 会话 1：来源探测 + 开关名称 → 来源 URL 的匹配策略
- 会话 2：爬取 + 下载 + 归因元数据管线
- 会话 3：集成到详情页 + 审查 / 手工修正错匹配

范围明确限定在 **"popular vendors"**（`data/vendors/` 下的 Cherry、Gateron、Akko、Kailh、Durock、JWK、NovelKeys、Outemu、TTC、Zeal），不触及 `other_vendors/`。

## 非目标（明确排除）

- 不在此功能内爬取规格数据（只爬图片）。
- 不做 AI 图像生成或 3D 渲染。
- 不重写现有的图片展示 UI（该工作属于功能 001）。
- 不处理 `other_vendors/` 存档。

## 约束（进入塑形阶段的硬性要求）

- **归因**：每张图片必须记录来源 URL + 站点名，像 ThereminGoat 数据那样公开致谢。
- **自托管**：下载到 `public/images/vendors/<Vendor>/<Switch>/`——不允许热链接。
- **robots.txt + 限速**：爬虫必须遵守；至少 1 req/sec，User-Agent 声明身份。
- **可审查**：错匹配必然发生；必须有人工抽查 / 覆盖机制，不能盲信爬虫输出。

## 框架陈述

> "如果我们能在 2-3 个会话内塑形并执行一个带归因的图片爬虫，
> 将 popular vendors 下 ~512 个开关的视觉覆盖率从 <1% 提升到足够让详情页'看起来完整'的水平，
> 我们将关闭与 switchesdb 的最大可见功能差距，并把 SwitchAtlas 从'数据表'转变为真正的'图谱'。"
